## 1. Профиль и условия пилота

- **Кто:** 1 владелец, несколько магазинов одной сети (multi-branch).
- **Режим:** SHADOW — RAOS работает **параллельно** существующей кассе клиента.
- **Что считаем:** учёт, склад, аналитика. **Фискальный чек RAOS НЕ выдаёт** (REGOS/KKM — #61/#62, Фаза-1).
- **Деньги:** 1-й месяц бесплатно → далее по тарифу.
- **Биллинг:** ручной (Payme/Click merchant ещё не подключён — #127/#128, дедлайн = старт пилота + 30 дней).
- **Резерв:** второй магазин держим в запасе на случай провала первого.

## 2. Этапы онбординга

Каждый этап: **действие → ответственный → verify (DoD) → ориентир по времени**.
Время — порядок величины, не обязательство.

### Этап 0 — Лид → квалификация → договорённость
- **Действие:** заявка с landing → созвон → подтвердить, что профиль подходит (multi-branch, готов к shadow), зафиксировать ожидания.
- **Ответственный:** CTO (+ Bekzod как PM).
- **Verify:** письменная договорённость о пилоте (условия, сроки, контактное лицо со стороны клиента, список филиалов).

### Этап 1 — Создание tenant (multi-branch)
- **Действие:** создать tenant владельца, завести **все филиалы** клиента. Создать учётки сотрудников по ролям (OWNER → MANAGER → CASHIER per branch).
- **Ответственный:** CTO (создание), Ibrat (если требуется доступ к админ-операциям/seed по #105).
- **Verify:** владелец логинится; видит все филиалы; роли применяются (RBAC); tenant изолирован (нет данных других tenant).
- **⚠️ Verify-шаги UI уточнить по реализации demo tenant #105** — не выдумывать, опереться на готовый демо-сценарий.

### Этап 2 — Импорт ассортимента (×N филиалов, done-for-you)
- **Действие:** собрать прайс/остатки клиента → импорт Excel/CSV по каждому филиалу. Это **миграция «под ключ»**, делаем мы, не клиент.
- **Ответственный:** CTO (данные + прогон), Ibrat (инструмент импорта — #104).
- **Verify:** товары/категории/штрихкоды загружены по всем филиалам; начальные остатки сходятся с тем, что дал клиент; выборочная сверка 10–15 позиций совпадает.
- **Зависимость:** #104 (Excel/CSV import) — Фаза-1; если ещё не готов, делать импорт через подтверждённый временный путь, согласовав с Ibrat.

### Этап 3 — Обучение персонала (1–2 часа)
- **Действие:** очно/онлайн показать кассирам и менеджеру: продажа, возврат, смена, склад, базовая аналитика владельцу. Telegram-бот для алертов владельца.
- **Ответственный:** CTO.
- **Verify:** кассир самостоятельно проводит тестовую продажу и возврат; менеджер открывает/закрывает смену; владелец получает Telegram-алерт.

### Этап 4 — Go-live SHADOW
- **Действие:** клиент начинает дублировать операции в RAOS параллельно своей кассе. Период shadow согласован (напр. 2–4 недели).
- **Ответственный:** CTO (контроль), клиент (ввод).
- **Verify:** ≥3 рабочих дня подряд данные вводятся по всем филиалам; ежедневная аналитика формируется; критичных ошибок (P0/P1 в `logs/errors-*.log` от этого tenant) нет.

### Этап 5 — Ручной биллинг, месяц 1
- **Действие:** месяц бесплатный — счёт не выставляем. Фиксируем дату старта = точка отсчёта T+30.
- **Ответственный:** CTO.
- **Verify:** дата старта пилота записана (см. §6); напоминание на T+25 о подключении merchant.

### Этап 6 — T+30: конверсия в тариф
- **Действие:** к T+30 merchant (#127/#128) должен быть готов → выставить тариф → авто-биллинг.
- **Ответственный:** CTO (merchant), Ibrat (тех. интеграция биллинга при необходимости).
- **Verify:** клиент на платном тарифе ИЛИ принято решение о продлении/выходе с зафиксированной причиной.

## 3. Что такое SHADOW-режим (границы)

**RAOS В shadow ДЕЛАЕТ:**
- учёт продаж и возвратов (нефискально);
- складской учёт и движения остатков по филиалам;
- аналитику (выручка, маржа, мёртвый сток, истечение срока годности);
- Telegram-алерты владельцу.

**RAOS В shadow НЕ ДЕЛАЕТ:**
- **не выдаёт фискальный чек** — официальная касса остаётся у клиента (REGOS/KKM = #61/#62, Фаза-1);
- не заменяет налоговую отчётность;
- не списывает деньги автоматически (биллинг ручной до T+30).

Это снимает регуляторный риск на время пилота и даёт показать ценность (аналитика/склад) без фискальной интеграции.

## 4. Сбор данных пилота (метрики успеха)

Заполнять в течение пилота — основа решения «конверсия / продление / выход».

| Метрика | Как снимаем | Целевой сигнал |
|---------|-------------|----------------|
| Дней активного ввода | по логам/аналитике tenant | ≥ согласованного shadow-периода |
| Кол-во филиалов в работе | админка | = всем заявленным |
| P0/P1 инциденты этого tenant | `logs/errors-*.log` + Tasks.md | 0 нерешённых |
| Ценность для владельца | интервью на выходе | называет ≥1 инсайт, который дал RAOS |
| Готовность платить | разговор на T+25..T+30 | да / условно / нет + причина |

## 5. Rollback / критерии остановки

- **P0 у tenant (потеря/искажение данных):** остановить ввод, эскалация Ibrat, разбор по `systematic-debugging`, не продолжать пока не закрыто.
- **Клиент не вводит данные ≥3 дней:** созвон, выяснить барьер; если не решается — переключиться на резервный второй магазин.
- **Merchant не готов к T+25:** продлить бесплатный период, зафиксировать новый T+30, эскалация по #127/#128.
- **Полный отказ клиента:** зафиксировать причину в §4, занести урок в ретро, активировать резервный магазин.

Откат безопасен: данные клиента — в его боевой кассе, RAOS shadow можно остановить без последствий для его операций.
